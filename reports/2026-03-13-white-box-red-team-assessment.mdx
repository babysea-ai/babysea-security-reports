# BabySea Public Red-Team Assessment Report

> **Report type:** Public sanitized security assessment summary  
> **Assessment date:** March 13, 2026  
> **Assessment mode:** Authorized white-box red-team assessment  
> **Systems tested:** BabySea production API and execution layer  
> **Regions tested:** US, EU, APAC  
> **Total attack attempts:** 300  
> **Public classification:** Sanitized for publication

This document is a public, sanitized version of BabySea's pre-launch red-team assessment. It is written for customers, partners, developers, and security reviewers who want to understand how BabySea tested its production execution layer before launch.

Operational identifiers in this report are anonymized. The original internal assessment used live regional test accounts and production endpoints, but API keys, account IDs, generation IDs, internal log rows, and reusable exploit payloads are intentionally not published.

This public report is not a certification, audit attestation, or guarantee of perfect security. It summarizes one authorized assessment within a defined scope and time window. It is structured as a security-evidence artifact: defined scope, methodology, control objectives, attack themes, expected failure modes, observed outcomes, evidence mapping, remediation status, limitations, and disclosure channels.

---

## Executive summary

BabySea ran an authorized white-box red-team assessment against its production API and execution layer before launch.

The test covered 300 attack attempts across three isolated regions: US, EU, and APAC. Each region received the same methodology with 100 attacks per region, allowing BabySea to compare behavior across regional infrastructure, application validation, authentication, rate limiting, database isolation, execution-state handling, and credit-ledger integrity.

The assessment was designed as a control-validation exercise, not a vanity scan. Each category mapped to a concrete failure mode that could matter to customers: unauthorized access, tenant-boundary failure, regional replay, schema bypass, SSRF, forged callbacks, protocol ambiguity, or credit-ledger corruption.

The assessment focused on six categories:

1. Authentication and authorization
2. Financial and state logic
3. Input validation and injection
4. Edge and protocol abuse
5. SSRF and file access
6. Webhook and infrastructure abuse

Every successful test generation produced a matching reserve -> charge pair. No orphaned reservations were found. No double charges were detected. No invalid refunds occurred. Account ownership, API key ownership, and generation ownership remained correctly linked.

The public conclusion is therefore bounded but strong: within the tested scope, BabySea's core execution invariants held under adversarial pressure.

## Why this assessment was necessary

BabySea is an execution control plane for generative media.

That means the platform is responsible for more than routing requests to model providers. It coordinates API keys, async image and video workloads, provider failover, credit reservation, credit settlement, webhooks, file delivery, usage logs, health checks, and regional execution.

In this type of system, the most dangerous bugs are usually not cosmetic UI failures. They are state failures.

A serious bug could become:

- a credit leak
- a duplicate charge
- a stuck reservation
- a refund race
- cross-account generation access
- forged provider webhook completion
- SSRF through user-supplied media URLs
- broken regional isolation
- provider abuse through malformed execution requests
- corrupted usage or billing records

The assessment therefore focused on execution-layer invariants, not only endpoint availability.

The key question was:

> Can hostile traffic corrupt BabySea's identity, tenancy, execution state, regional isolation, webhook state, file access, or credit ledger?

## Assessment model

This was a white-box red-team assessment.

Before the attack suite was designed, the tester was given time to inspect BabySea's application structure, API routes, database schema, billing model, generation lifecycle, regional deployment pattern, and infrastructure boundaries.

The goal was not to simulate a generic internet scanner. The goal was to simulate an attacker who understood the system and targeted the seams most likely to break a generative-media execution layer.

### Scope design

| Scope area | Public description |
| --- | --- |
| Regions | US, EU, APAC |
| Methodology | Identical methodology in every region |
| Accounts | Dedicated regional test accounts |
| API keys | Dedicated regional test API keys |
| Workloads | Real API requests and real generation lifecycle paths |
| Data plane | Production execution layer |
| Result analysis | API responses, application behavior, credit-ledger integrity, asset records, and regional consistency |

### Methodology constraints

The test followed these public constraints:

- Exactly 100 attack attempts were executed per region.
- The same category structure was used in every region.
- Attack attempts targeted real API surfaces, not only synthetic mocks.
- Financial and execution-state tests used real generation lifecycle paths.
- Public reporting excludes reusable exploit scripts and sensitive operational identifiers.

### Governance and evidence handling

This public report is intentionally written in a format that security, compliance, and engineering teams can evaluate without access to BabySea's internal systems.

| Area | Public handling |
| --- | --- |
| Authorization | The assessment was authorized and scoped before execution. |
| Environment | Testing targeted dedicated regional test accounts and production execution paths. |
| Evidence sources | Assessment outputs were cross-checked against API responses, authenticated request logs, credit-ledger records, and generated asset records. |
| Sensitive material | API keys, account IDs, generation IDs, internal row identifiers, raw payload scripts, and reusable exploit automation are excluded from publication. |
| Result discipline | Results are described as observed outcomes within a defined test window, not as a universal guarantee. |
| Assurance boundary | This is an internal authorized white-box assessment summary, not an independent third-party audit or formal compliance attestation. |

The purpose of this evidence model is to make the report useful to security reviewers while avoiding disclosure that would materially help an attacker reproduce the attack suite.

## High-level architecture under test

The public assessment covered BabySea's external execution layer and the state systems behind it.

```text
Client request
  -> regional BabySea API
  -> API key authentication
  -> account and scope resolution
  -> input validation
  -> rate and concurrency controls
  -> credit reservation
  -> provider selection and failover
  -> provider execution
```

The test also validated region-specific behavior across US, EU, and APAC. The purpose was to ensure that a security property in one region did not silently fail in another.

## Result classification model

BabySea classified each attempt by observed behavior and security impact.

| Classification | Meaning |
| --- | --- |
| Directly blocked | The request was rejected by authentication, validation, method restrictions, runtime behavior, or protocol handling before unsafe processing. |
| Expected-safe behavior | The request reached a legitimate code path but produced a safe outcome, such as scoped `404`, lifecycle `409`, rate-limit `429`, or a response tied only to the authenticated account. |
| Unexpected result | The request produced behavior outside the expected security model, such as unauthorized access, unsafe state transition, credit inconsistency, or sensitive disclosure. |
| Runtime error | The test or target path errored in a way that prevented reliable security classification. |

Severity ratings were reserved for unexpected results. Because the 300-attempt attack suite produced no unexpected results, no critical, high, medium, or low findings were assigned within the tested scope.

## CAT1: Authentication and authorization

### Objective

Validate that BabySea derives identity and account ownership only from verified credentials, not from user-controlled headers, route parameters, region confusion, or malformed token material.

### Public attack themes

CAT1 tested:

- missing authentication
- malformed Bearer values
- empty tokens
- invalid token formats
- token injection attempts
- cross-region token replay
- random keys with valid-looking prefixes
- API keys supplied through unsafe channels
- BOLA and IDOR attempts against generation access
- destructive operations against non-owned generation IDs
- fake JWT-style authentication material
- spoofed account headers
- oversized authorization material
- IP spoofing attempts through forwarding headers

### Expected failure mode

A vulnerable system might accept a forged or malformed credential, trust a caller-supplied account header, allow a regional API key in the wrong region, or expose another account's generation data by ID.

### Public result

No unexpected findings were observed.

BabySea's API resolved identity from verified API keys, rejected invalid authentication material, did not trust spoofed account headers, and did not expose non-owned generation records through tested object-access paths.

## CAT2: Financial and state logic

### Objective

Validate that hostile concurrency and malformed pricing inputs cannot corrupt BabySea's generation state or credit ledger.

This was the highest-priority category because BabySea manages async workloads with credit reservation and settlement.

### Public attack themes

CAT2 tested:

- concurrent generation pressure
- TOCTOU-style credit race attempts
- heavier burst generation attempts
- concurrent cancel/refund attempts
- spoofed generation IDs
- cross-account cancel attempts
- negative, zero, or oversized pricing inputs
- estimate-versus-generate mismatch attempts
- delete-then-cancel sequencing
- concurrent estimate and generation behavior
- invalid video duration and resolution boundaries

### Expected failure mode

A vulnerable system might allow more generations than the available balance supports, double-refund a single generation, charge the wrong account, refund a nonexistent generation, or calculate negative/zero cost for a billable workload.

### Public result

No unexpected findings were observed.

The most important result was credit-ledger integrity.

| Region | Reserves | Charges | Result |
| --- | ---: | ---: | --- |
| US | 10 | 10 | Balanced |
| EU | 10 | 10 | Balanced |
| APAC | 11 | 11 | Balanced |

The APAC region completed one additional generation during a rate-limit timing window, but the ledger still balanced. Each successful generation had a matching reserve and charge. No orphaned reservations, double charges, or invalid refunds were observed.

### Public conclusion

The tested financial invariant held:

```text
successful generation -> exactly one reserve -> exactly one charge
```

For BabySea, this was the most important assessment outcome.

## CAT6: Webhook and infrastructure abuse

### Objective

Validate that forged provider callbacks, replay attempts, infrastructure probes, and concurrent route pressure cannot create false state transitions or exhaust critical infrastructure paths.

In BabySea, webhooks are part of the async execution state machine. A forged webhook is not merely a fake notification. It could become a false completion, false failure, false refund, or corrupted generation record if not handled correctly.

### Public attack themes

CAT6 tested:

- forged provider webhook callbacks
- missing or empty webhook signatures
- stale and future timestamp behavior
- oversized webhook body handling
- cron endpoint probing
- unauthenticated infrastructure route probing
- information disclosure checks
- burst rate-limit behavior
- rate-limit scope validation
- webhook injection probes
- DLQ stress behavior
- concurrent multi-route pressure

### Expected failure mode

A vulnerable system might accept a forged provider completion, process a replayed webhook, expose sensitive version or environment data, allow unauthenticated cron execution, or fail under concurrent route pressure.

### Public result

No unexpected findings were observed.

Forged callbacks and infrastructure probes did not create unsafe state transitions during the assessment.

## Cross-region consistency

The assessment intentionally repeated the same methodology across US, EU, and APAC.

This was necessary because multi-region systems can drift in subtle ways:

- different environment variables
- different database schema state
- different edge behavior
- different rate-limit timing
- different region-specific provider behavior
- different secrets or webhook configuration
- different route deployment state

### Public observation

Five timing-dependent differences were observed across regions. These were related to rate-limit window alignment and generation completion timing. They did not create security gaps and did not affect credit integrity.

### Public conclusion

The regions behaved consistently at the policy and invariant level. Timing differed in some cases, but identity, tenancy, validation, and credit-ledger invariants held.

## Edge-security configuration note

Two days after the assessment concluded, BabySea identified that the planned Cloudflare WAF/API Shield policy set was not fully enforcing as intended during the test window. Traffic still traversed normal edge and runtime infrastructure, but the custom edge-security controls were incomplete.

This is reported here in the interest of transparency. It does not invalidate the assessment results. It clarifies them.

The 300-attempt suite therefore relied more heavily on BabySea's application, authentication, validation, rate-limit, and credit-ledger controls than originally intended, without the full planned custom edge-security policy set assisting. Those inner controls handled the assessment without unexpected findings, and the credit ledger remained balanced under adversarial concurrency.

After the discovery, BabySea completed the edge-security hardening work and treats those controls as defense-in-depth on top of the application controls validated here, not in place of them.

The public lesson:

> Edge controls should reduce risk and block hostile traffic earlier. They should never be the only layer protecting application state. The application, database, and billing invariants must hold on their own.

BabySea's architecture is designed so that the inner invariants still matter even when edge assumptions are incomplete. The incomplete edge-policy enforcement during this assessment provided bounded evidence, within the tested scope, that this property held.

## Public limitations

This report is intentionally scoped.

It does not claim that BabySea is perfectly secure. No serious security report should make that claim.

The assessment means:

- the tested attack suite did not produce unexpected findings
- the tested regions preserved the expected invariants
- the tested credit-ledger behavior remained balanced
- the tested SSRF, auth, injection, protocol, and webhook paths were blocked or safely handled
- the tested application-layer controls did not depend solely on edge filtering

The report does not mean:

- every possible vulnerability class was tested
- every future code change inherits this result
- every provider-side behavior is under BabySea's control
- every possible edge-case payload is impossible
- this public report replaces independent third-party security review

BabySea will continue to test the execution layer as the product surface expands.
